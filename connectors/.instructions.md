# Connector Page Rules

Use `connectors/_TEMPLATE.md` as the structural reference.

## Section Order

1. H1 title → Metadata line (Tier · Type · Free ingestion) → Contents → Overview → Licensing Benefits → Tables and Rationale → Example Detections → MITRE Detection Strategies → MCSB Control Mapping → [Optional sections] → Notes → Tools → References → Footer

## Contents / TOC Editing Rule

- **Preserve existing TOC style** on each page. If a page already uses nested bullets, keep nested bullets; if it uses a flat list, keep flat list.
- When adding `## MITRE Detection Strategies`, do the **smallest TOC change possible**: add the missing link only, without flattening, reordering, or re-indenting unrelated items.
- Do not rewrite a page-wide TOC format unless explicitly requested.

## Table Formats

**Tables and Rationale** columns: `| Table | Description | Retention Recommendation | Rationale | Forensic Value | Example Detection |`
- Retention default: `Analytics: 90d / Lake: 365d`
- Forensic Value: action-oriented phrasing ("Reconstruct…", "Prove…", "Detect…")
- Table names in backticks

**Per-event-ID tier tables** (when a connector page enumerates individual Event IDs under "Key Events by Tier" or equivalent) columns: `| Event ID | Description | Minimal | Common | Full | Rationale | Forensic Value | MITRE | Example Detection |`
- The `Rationale` and `Example Detection` cells **must not** contain MCSB control IDs (e.g. `MCSB IM-1.`, `MCSB PA-1.`, `MCSB LT-3.`) or MITRE technique IDs (e.g. `MITRE T1136.001.`, trailing `(T1110)`, `(T1059.001)`). MCSB belongs in the `MCSB Control Mapping` section; MITRE technique/strategy references belong in the new `MITRE` column and the dedicated `MITRE Detection Strategies` section.
- The `MITRE` column links to the page's own `## MITRE Detection Strategies` section using `[N mappings](#mitre-detection-strategies)` (or `[1 mapping](#mitre-detection-strategies)`), where `N` is the count of distinct `(CurrentTechId, DetId)` pairs in your local MITRE mapping detail export whose `Platform` = the connector's platform and whose `EventCode` field contains the row's Event ID (regex `\bID\b`, so comma-separated lists like `EventCode=4624, 4648` match both). When no row matches, render `—`. Do **not** inline individual technique or strategy links into the per-event cell — the strategies section is the canonical list.

**Example Detections** columns: `| Detection | Table(s) / Event ID(s) | MITRE ATT&CK | Detection Strategy | Description |`
- MITRE format: full technique ID `T1110.003` linked to `https://attack.mitre.org/techniques/T1110/003/`
- Detection Strategy format: `[DET####](https://attack.mitre.org/detectionstrategies/DET####/) — [Strategy name]`, or `— *(no published strategy)*` if none

**MITRE Detection Strategies** columns:
- Default: `| Technique | Detection Strategy |`
- Raw-channel exception only: `| Technique | Detection Strategy | MITRE Log Sources (<Platform>) |` or `| Technique | Detection Strategy | Connector Evidence (<Platform>) |`

- **Connector classification.** Use the 3rd column only for pages that are intentionally maintained as raw-channel mappings (currently `windows-security-events`, `windows-forwarded-events`, `syslog-linux`, `dns-security-logs`, `iis-web-server-logs`).
- For cloud/service connectors and portal-normalised connectors, use the 2-column format and add a `[!NOTE]` explaining why the 3rd column is omitted.
- If a page already uses the 2-column format, keep it 2-column unless explicitly told to change.
- Source of truth: a local MITRE mapping detail export (one row per `CitedTech, Strategy, Analytic, LogSource`; built from the MITRE STIX `enterprise-attack` bundle by your local MITRE-mapping build process; follows `revoked-by` chain). Keep a matching local summary export as well. Older legacy mapping exports are superseded. If local exports are unavailable, derive the mapping directly from the MITRE STIX bundle instead of assuming any machine-specific cache path or filename.
- The third column header substitutes the connector's primary platform (e.g. `(Windows)` for Windows event sources, `(Linux)` for syslog, `(IaaS)` for cloud control-plane pages).
- Use **MITRE Log Sources (<Platform>)** only when the connector's native evidence is close enough to MITRE's published source names that a verbatim rendering remains clear.
- Use **Connector Evidence (<Platform>)** when a verbatim rendering would show another vendor's table names or product labels on this connector page. In that mode, translate the strategy into the connector's own tables, operations, and telemetry categories instead of reproducing foreign vendor source names.
- For **MITRE Log Sources** mode, filter the detail CSV by `CitedTechId`, `DetId` and `Platform = <connector's platform>`. Group rows by `Channel` and concatenate `EventCode` values within each group. Render `` `Channel`: <codes> `` joined with ` · `. Example: `` `Security`: 4624, 4648 · `Sysmon`: 1, 3, 22 ``.
- For **Connector Evidence** mode, write the analogous evidence directly in the current connector's terms: actual table names, operation families, or telemetry types available on the page. Preserve MITRE's analytic intent, but avoid implying a literal 1:1 table-name match across vendors.
- When no row matches the connector's platform in **MITRE Log Sources** mode, render: `*MITRE has not published a <Platform> analytic for this strategy*`.
- When the platform rows contain only non-`Security` channels (only `Sysmon` / `PowerShell` / `System`), still list them verbatim and append `*(no \`Security\`-channel analytic published)*`.
- If a cited technique was revoked, annotate as `Txxxx.xxx — [Name] *(revoked → Tyyyy.yyy)*` and link the strategy attached to the current technique.
- Include a `[!NOTE]` callout explaining which third-column mode is in use. For **MITRE Log Sources** mode, use the existing "log sources are verbatim from MITRE" wording. For **Connector Evidence** mode, explain that MITRE's published strategy was translated into the analogous evidence available in this connector's own tables because verbatim foreign-vendor source names would be misleading here.
- Include a `[!NOTE]` legacy-ID callout only when at least one row uses a revoked ID.
- Only mark a technique as "no published strategy" (in the `Example Detections` table) if neither the legacy nor the current (post-`revoked-by`) technique appears in your local MITRE mapping summary export.
- ATT&CK for ICS techniques may legitimately have no MITRE Detection Strategy yet; in those cases use `— *(no published strategy)*`.

**MCSB Control Mapping** columns: `| MCSB Control | Relevance |`

**Tools** columns: `| Tool | Type | Purpose | Source | Guide |`
